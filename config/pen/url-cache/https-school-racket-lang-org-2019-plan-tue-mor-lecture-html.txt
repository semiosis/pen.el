yesterday:

a sequence where each element of <span class="RktSym">seq</span> is bound to the

our implementation of this macro fails to present this abstraction:
instead, it exposes its implementation to clients.</p><p><div class="SIntrapara">First, we expose that we are using <span class="RktSym"><a href="http://docs.racket-lang.org/reference/pairs.html#%28def._%28%28lib._racket%2Fprivate%2Fmap..rkt%29._map%29%29" class="RktValLink" data-pltdoc="x">map</a></span> in the implementation

the client is expected to understand the details of the

new abstractions by verifying pre-conditions. Some of these

<span class="RktSym">elem-name</span> is not a variable in the <span class="RktSym">robust-for/list3</span>
program. Instead, it is a pattern variable in the first pattern
clause. Thus, the name is only valid inside of syntax templates. Thus,
to refer to the syntax provided, we must use
<span class="RktSym"><a href="http://docs.racket-lang.org/reference/stx-patterns.html#%28form._%28%28lib._racket%2Fprivate%2Fstxcase-scheme..rkt%29._syntax%29%29" class="RktStxLink" data-pltdoc="x">syntax</a></span>/<span class="RktInBG"><span class="hspace"></span><span class="RktIn">#'</span><span class="hspace"></span></span>.</p><p>Next, look at the next line. <span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._raise-syntax-error%29%29" class="RktValLink" data-pltdoc="x">raise-syntax-error</a></span> is a special
version of <span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._error%29%29" class="RktValLink" data-pltdoc="x">error</a></span> that is optimized for syntactic errors. Like
<span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._error%29%29" class="RktValLink" data-pltdoc="x">error</a></span>, it takes as arguments the name of the erroring
procedure and a message. However, it also takes two arguments that
evaluate to syntax. The first is the "large" piece of syntax that
contains the error; in this case, the offending usage of
<span class="RktSym">robust-for/list3</span>. The second is the "small" piece of syntax
that is at fault; in this case, the actual syntax given for
<span class="RktSym">elem-name</span>. It is far better to use
<span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._raise-syntax-error%29%29" class="RktValLink" data-pltdoc="x">raise-syntax-error</a></span> than to use <span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._error%29%29" class="RktValLink" data-pltdoc="x">error</a></span>, because Racket
IDEs (like DrRacket) will hone their error highlighting with this
extra information.</p><p>However, <span class="RktSym">robust-for/list3</span> still has a problem. By using
<span class="RktSym"><a href="http://docs.racket-lang.org/reference/when_unless.html#%28form._%28%28lib._racket%2Fprivate%2Fletstx-scheme..rkt%29._unless%29%29" class="RktStxLink" data-pltdoc="x">unless</a></span> in the template position of the clause, we do not
cooperate with <span class="RktSym"><a href="http://docs.racket-lang.org/syntax/Parsing_Syntax.html#%28form._%28%28lib._syntax%2Fparse..rkt%29._syntax-parse%29%29" class="RktStxLink" data-pltdoc="x">syntax-parse</a></span> to identify syntax that does not
match the pattern expected by the macro. This is a subtle point. It is
the job of <span class="RktSym"><a href="http://docs.racket-lang.org/syntax/Parsing_Syntax.html#%28form._%28%28lib._syntax%2Fparse..rkt%29._syntax-parse%29%29" class="RktStxLink" data-pltdoc="x">syntax-parse</a></span> to determine if the input syntax
satisfies this clause. If we perform extra tests after it passes the
pattern, then we are obscuring the conditions from
<span class="RktSym"><a href="http://docs.racket-lang.org/syntax/Parsing_Syntax.html#%28form._%28%28lib._syntax%2Fparse..rkt%29._syntax-parse%29%29" class="RktStxLink" data-pltdoc="x">syntax-parse</a></span>. In this particualr case, it is not an issue,

being at fault (unlike our use of <span class="RktSym"><a href="http://docs.racket-lang.org/reference/exns.html#%28def._%28%28quote._~23~25kernel%29._raise-syntax-error%29%29" class="RktValLink" data-pltdoc="x">raise-syntax-error</a></span>). This

more minute specifications about the form of particular syntactic
elements: <span style="font-style: italic">syntax classes</span>.</p><h4>5.4<tt>&nbsp;</tt><a name="(part._.Classy_.Syntax)"></a>Classy Syntax</h4><p><span class="RktSym"><a href="http://docs.racket-lang.org/syntax/Parsing_Syntax.html#%28form._%28%28lib._syntax%2Fparse..rkt%29._syntax-parse%29%29" class="RktStxLink" data-pltdoc="x">syntax-parse</a></span> clause patterns supports annotations on pattern
variables that specify which syntactic category (i.e. class) they
belong to. The notation for these annotations is
<span class="RktSym">pattern-variable:syntax-class</span>. Its pattern matching algorithm

are not satisfied, then it will generate a high-quality error message,

Here&rsquo;s a
very brief summary of a few points.</p><p>How do we expose attributes of a syntax class if there are multiple
variants that define different sub-pattern variables? The
<span class="RktSym"><a href="http://docs.racket-lang.org/syntax/stxparse-specifying.html#%28form._%28%28lib._syntax%2Fparse..rkt%29._define-syntax-class%29%29" class="RktStxLink" data-pltdoc="x">define-syntax-class</a></span> form allows an explicit list of
attributes to be named outside of any pattern and all patterns are
checked to ensure they bind these.</p><p>Are attributes always bound to sub-pieces of syntax, like in our
examples? Absolutely not. We can use attributes to bind syntax that
the class synthesized via syntax template substitution. We can even
use attributes to hold arbitrary information and data structures, not
just syntax objects.</p><p>Can we define syntax classes where the pattern is not surrounded by
parentheses? Yes, these are called "splicing syntax classes".</p><p>There are many many other subtle points and extensions that make
syntax classes yet another powerful weapon in the macrologist&rsquo;s
