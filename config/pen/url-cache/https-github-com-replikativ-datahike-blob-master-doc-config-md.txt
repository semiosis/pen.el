    <article class="markdown-body entry-content container-lg" itemprop="text"><h1 dir="auto"><a id="user-content-datahike-database-configuration" class="anchor" aria-hidden="true" href="#datahike-database-configuration"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Datahike database configuration</h1>
<p dir="auto">At database creation <em>Datahike</em> supports features that can be configured based on the application's requirements. As of version <code>0.2.0</code> configuration for the <a href="#storage-backend">storage backend</a>, the <a href="#schema-flexibility">schema flexibility</a>, and <a href="#historical-data">time variance</a> is supported.  Be aware: all these features can be set at database creation but can not be changed afterwards. You can still migrate the data to a new configuration.</p>
<h2 dir="auto"><a id="user-content-configuration" class="anchor" aria-hidden="true" href="#configuration"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Configuration</h2>
<p dir="auto">Configuring <em>Datahike</em> is now possible via the <a href="https://github.com/weavejester/environ">environ library made by weavejester</a>. You can use environment variables, java system properties and passing a config-map as argument.</p>
<p dir="auto">The sources are resolved in following order:</p>
<ol dir="auto">
<li>Environment variables</li>
<li>Java system properties</li>
<li>Argument to load-config</li>
</ol>
<p dir="auto">That means passing a config as argument overwrites java system properties and using java system properties overwrite environment variables etc. Currently the configuration map looks like this per default:</p>
<div class="highlight highlight-source-clojure notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="{:store              {:backend  :mem        ;keyword
                      :id        &quot;default&quot;} ;string

 :schema-flexibility :write                 ;keyword
 :keep-history?      true}                  ;boolean"><pre>{<span class="pl-c1">:store</span>              {<span class="pl-c1">:backend</span>  <span class="pl-c1">:mem</span>        <span class="pl-c"><span class="pl-c">;</span>keyword</span>
                      <span class="pl-c1">:id</span>        <span class="pl-s"><span class="pl-pds">"</span>default<span class="pl-pds">"</span></span>} <span class="pl-c"><span class="pl-c">;</span>string</span>

<p dir="auto">If you are using a backend different from the builtins <code>:mem</code> or <code>:file</code>, please have a look at the README in the corresponding Github repository. The configuration is outsourced to the backends so you will find the configuration documentation there. An example for <code>:mem</code>, <code>:file</code>, and <code>:jdbc</code>-backend you can see below. Please refer to the documentation of the <a href="https://github.com/weavejester/environ">environ library</a> on how to use it. <h2 dir="auto"><a id="user-content-storage-backend" class="anchor" aria-hidden="true" href="#storage-backend"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Storage Backend</h2>
<p dir="auto">Each backend needs a different set of provided parameters. See definition
<a href="#storage-backend">below</a> for further information. For simple and fast creation
you can simply use the defaults which creates an in-memory database with ID <code>"default"</code>, write schema flexibility, and history support:</p>
<div class="highlight highlight-source-clojure notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="(require '[datahike.api :as d])

<h2 dir="auto"><a id="user-content-name" class="anchor" aria-hidden="true" href="#name"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Name</h2>
<p dir="auto">By default <em>Datahike</em> generates a name for your database for you. If you want to set
the name yourself just set a name for it in your config. It helps to specify the
database you want to use, in case you are using multiple <em>Datahike</em> databases in
your application (to be seen in datahike-server).</p>
<h2 dir="auto"><a id="user-content-schema-flexibility" class="anchor" aria-hidden="true" href="#schema-flexibility"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Schema Flexibility</h2>
<p dir="auto">By default the <em>Datahike</em> api uses a schema on <code>:write</code> approach with strict value
types that need to be defined in advance. If you are not sure how your data
model looks like and you want to transact any kind of data into the database you
can set <code>:schema-flexibility</code> to <code>read</code>. You may add basic schema definitions like <code>:db/unique</code>,
<code>:db/cardinality</code> or <code>db.type/ref</code> where these kind of structure is needed.</p>
<div class="highlight highlight-source-clojure notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="(require '[datahike.api :as d])

(<span class="pl-en">d/create-database</span> {<span class="pl-c1">:schema-flexibility</span> <span class="pl-c1">:read</span>})</pre></div>
<p dir="auto">Have a look at the <a href="/replikativ/datahike/blob/master/doc/schema.md">schema documentation</a> for more information.</p>
<h2 dir="auto"><a id="user-content-historical-data" class="anchor" aria-hidden="true" href="#historical-data"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Historical Data</h2>
<p dir="auto">Datahike has the capability to inspect and query historical data within temporal
indices. If your application does not require any temporal data, you may
set <code>:keep-history?</code> to <code>false</code>.</p>
<div class="highlight highlight-source-clojure notranslate position-relative overflow-auto" data-snippet-clipboard-copy-content="(require '[datahike.api :as d])
(d/create-database {:keep-history? true})"><pre>(<span class="pl-k">require</span> '[datahike.api <span class="pl-c1">:as</span> d])
(<span class="pl-en">d/create-database</span> {<span class="pl-c1">:keep-history?</span> <span class="pl-c1">true</span>})</pre></div>
<p dir="auto">Be aware: when deactivating the temporal index you may not use any temporal databases like <code>history</code>, <code>as-of</code>, or
<code>since</code>.</p>
<p dir="auto">Refer to the <a href="/replikativ/datahike/blob/master/doc/time_variance.md">time variance documentation</a> for more information.</p>
<h2 dir="auto"><a id="user-content-deprecation-notice" class="anchor" aria-hidden="true" href="#deprecation-notice"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Deprecation Notice</h2>
<p dir="auto">Starting from version <code>0.3.0</code> it is encouraged to use the new hashmap configuration since it is more flexible than the previously used URI scheme. Datahike still supports the old configuration so you don't need to migrate yourself. The differences for the configuration are as following:</p>
<ul dir="auto">
<li>optional parameters are added in the configuration map instead of optional parameters</li>
<li><code>:temporal-index</code> renamed to <code>:keep-history?</code></li>
<li><code>:schema-on-read</code> renamed to <code>:schema-flexibility</code> with values <code>:read</code> and <code>:write</code></li>
<li>store configuration for backends moved into <code>:store</code> atttribute</li>
<li><code>:initial-tx</code> also added as attribute in configuration</li>
<li>the store configuration is now more flexible, so it fits better with its backends</li>
<li>all backend configuration remains the same except for <code>:mem</code></li>
<li>naming attribute for <code>:mem</code> backend is moved to <code>:id</code> from <code>:host</code> or <code>:path</code></li>
<li>optional <code>clojure.spec</code> validation has been added</li>
</ul>
