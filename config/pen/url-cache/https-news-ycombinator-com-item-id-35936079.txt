                  <div class="commtext c00">Sixel has the one advantage of being mplemented in xterm and a modest number of other terminals.  Otherwise, it&#x27;s a pretty bad format: Inefficient. Unclear and inconsistently implemented specification. All images have to be a multiple fof 6 pixel rows, which may not align with either image height or character height.<p>Some terminal implement some other protocols, but attempts to specify a standard have failed. There are some tricky issues, such as: When does an image or part of an image get erased? Can you write text on top of an image and if so how are they aligned? What happens if you write an image on top of existing text? On top of an existing image? How does scrolling affect things? What happens to the image on window resize or zoom? Can you reliably update part of an image?<p>DomTerm (<a href="https:&#x2F;&#x2F;domterm.org" rel="nofollow">https:&#x2F;&#x2F;domterm.org</a>) supports images in two ways:
For sixels, it creates a canvas element as an &quot;underlay&quot;. As it is earlier in he rendering order, following text can be written on top of the image.<p>DomTerm also supports general HTML, including IMG elements.  (Of course it is safety-scrubbed.) This is quite flexible, but it doesn&#x27;t overlap regular text (without some contortions). DomTerm allows different lines to have different height, which is beneficial for &quot;printing&quot; an image without having to pad the output to an integral number of rows.<p>Neither of these DomTerm protocls are suitable for a portable protocol that can be widely implemented. Two other protocols that I&#x27;m aware of are the iTerm2 protocol and the Kitty protocol. Both are bit on the complicated side, but a subset of one of those might be an acceptable &quot;standard&quot;.</div>

                  <div class="commtext c00">When this tech was invented it was less relevant than it is now. For the following reasons:<p>- It required special expensive terminals, the VT330 (mono) or VT340. Those were not ubiquitous, the VT320 was the bread and butter type. Later VTx20 models also didn&#x27;t gain support for it (usually the lower range models got premium features from the previous ranges like multiple connections &#x2F; pages but this one didn&#x27;t &#x27;trickle down&#x27;). So there wasn&#x27;t so much incentive for developers to support it as the installed base was very low.<p>- It required a long time to send all the information, because terminals were still on slow serial connections. The VT320 could only do up to 19200 bps which was not even super fast for text at 80x25! Sixel images would take ages to transfer. Even though we had different expectations of response speeds then, sixels were slow by the standard of the day.<p>- Colour highlighting (or even monochrome &quot;bold&quot;&#x2F;ultrabright) was not even common in most commandline tools back then, even LS. Hell, on HP-UX in the 90s I couldn&#x27;t even use the cursor keys in the default shell for command history and line editing! Mod cons we take for granted now were yet to be invented then.<p>- Thumbnail processing as the parent suggests would take a lot of space and processing speed. My quota at college was a few megabytes. In fact I hardly had any images. I think I had just one, a nice Enterprise-D render that I used as wallpaper. It&#x27;s not like we had thousands of images to sift through like we do today. Of course the web changed that rapidly but at that time text terminals were already on the way out.<p>Some apps used it for stuff that was worth the wait. Like the print preview in word perfect. But many apps didn&#x27;t bother because the above constraints just made them infeasible. Sixels only really took off after their original terminals were long gone. And really in this day and age we should be able to come up with something better.</div>
