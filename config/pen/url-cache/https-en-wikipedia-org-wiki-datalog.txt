<p><b>Datalog</b> is a <a href="/wiki/Declarative_programming" title="Declarative programming">declarative</a> <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a> language.  While it is syntactically a subset of <a href="/wiki/Prolog" title="Prolog">Prolog</a>, Datalog generally uses a bottom-up rather than top-down evaluation model.  This difference yields significantly different behavior and properties from <a href="/wiki/Prolog" title="Prolog">Prolog</a>.  It is often used as a <a href="/wiki/Query_language" title="Query language">query language</a> for <a href="/wiki/Deductive_database" title="Deductive database">deductive databases</a>. Datalog has been applied to problems in <a href="/wiki/Data_integration" title="Data integration">data integration</a>, <a href="/wiki/Computer_network" title="Computer network">networking</a>, <a href="/wiki/Program_analysis" title="Program analysis">program analysis</a>, and more.
</p>
<meta property="mw:PageProp/toc" />
<h2><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=1" title="Edit section: Example"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A Datalog program consists of <i>facts</i>, which are statements that are held to be true, and <i>rules</i>, which say how to deduce new facts from known facts. For example, here are two facts that mean <i>xerces is a parent of brooke</i> and <i>brooke is a parent of damocles</i>:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">parent</span><span class="p">(</span><span class="s s-Atom">xerces</span><span class="p">,</span> <span class="s s-Atom">brooke</span><span class="p">).</span>
<span class="nf">parent</span><span class="p">(</span><span class="s s-Atom">brooke</span><span class="p">,</span> <span class="s s-Atom">damocles</span><span class="p">).</span>
</pre></div>
<p>The names are written in lowercase because strings beginning with an uppercase letter stand for variables. Here are two rules:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">ancestor</span><span class="p">(</span><span class="nv">X</span><span class="p">,</span> <span class="nv">Y</span><span class="p">)</span> <span class="p">:-</span> <span class="nf">parent</span><span class="p">(</span><span class="nv">X</span><span class="p">,</span> <span class="nv">Y</span><span class="p">).</span>
<span class="nf">ancestor</span><span class="p">(</span><span class="nv">X</span><span class="p">,</span> <span class="nv">Y</span><span class="p">)</span> <span class="p">:-</span> <span class="nf">parent</span><span class="p">(</span><span class="nv">X</span><span class="p">,</span> <span class="nv">Z</span><span class="p">),</span> <span class="nf">ancestor</span><span class="p">(</span><span class="nv">Z</span><span class="p">,</span> <span class="nv">Y</span><span class="p">).</span>
</pre></div>
<p>The <code>:-</code> symbol is read as "if", and the comma is read "and", so these rules mean:
</p>
<ul><li><i>X is an ancestor of Y if X is a parent of Y.</i></li>
<li><i>X is an ancestor of Y if X is a parent of some Z, and Z is an ancestor of Y.</i></li></ul>
<p>The meaning of a program is defined to be the set of all of the facts that can be deduced using the initial facts and the rules. This program's meaning is given by the following facts:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">parent</span><span class="p">(</span><span class="s s-Atom">xerces</span><span class="p">,</span> <span class="s s-Atom">brooke</span><span class="p">).</span>
<span class="nf">parent</span><span class="p">(</span><span class="s s-Atom">brooke</span><span class="p">,</span> <span class="s s-Atom">damocles</span><span class="p">).</span>
<span class="nf">ancestor</span><span class="p">(</span><span class="s s-Atom">xerces</span><span class="p">,</span> <span class="s s-Atom">brooke</span><span class="p">).</span>
<span class="nf">ancestor</span><span class="p">(</span><span class="s s-Atom">brooke</span><span class="p">,</span> <span class="s s-Atom">damocles</span><span class="p">).</span>
<span class="nf">ancestor</span><span class="p">(</span><span class="s s-Atom">xerces</span><span class="p">,</span> <span class="s s-Atom">damocles</span><span class="p">).</span>
</pre></div>
<p>Some Datalog implementations don't deduce all possible facts, but instead answer <i>queries</i>:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="s s-Atom">?-</span> <span class="nf">ancestor</span><span class="p">(</span><span class="s s-Atom">xerces</span><span class="p">,</span> <span class="nv">X</span><span class="p">).</span>
</pre></div>
<p>This query asks: <i>Who are all the X that xerces is an ancestor of?</i> For this example, it would return <i>brooke</i> and <i>damocles</i>.
</p>
<h2><span class="mw-headline" id="Comparison_to_relational_databases">Comparison to relational databases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=2" title="Edit section: Comparison to relational databases"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>

<h2><span class="mw-headline" id="Syntax">Syntax</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=3" title="Edit section: Syntax"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>A Datalog program consists of a list of <i>rules</i> (<a href="/wiki/Horn_clause" title="Horn clause">Horn clauses</a>).<sup id="cite_ref-FOOTNOTECeriGottlobTanca1989146_1-0" class="reference"><a href="#cite_note-FOOTNOTECeriGottlobTanca1989146-1">&#91;1&#93;</a></sup> If <i>constant</i> and <i>variable</i> are two <a href="/wiki/Countable_set" title="Countable set">countable</a> sets of constants and variables respectively and <i>relation</i> is a countable set of <a href="/wiki/Predicate_variable" title="Predicate variable">predicate symbols</a>, then the following <a href="/wiki/BNF_grammar" class="mw-redirect" title="BNF grammar">BNF grammar</a> expresses the structure of a Datalog program:
</p>
<div class="mw-highlight mw-highlight-lang-bnf mw-content-ltr" dir="ltr"><pre><span></span><span class="p">&lt;</span><span class="nc">program</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">rule</span><span class="p">&gt;</span> <span class="p">&lt;</span><span class="nc">program</span><span class="p">&gt;</span> | &quot;&quot;
<span class="p">&lt;</span><span class="nc">rule</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">atom</span><span class="p">&gt;</span> &quot;:-&quot; <span class="p">&lt;</span><span class="nc">atom-list</span><span class="p">&gt;</span> &quot;.&quot;
<span class="p">&lt;</span><span class="nc">atom</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">relation</span><span class="p">&gt;</span> &quot;(&quot; <span class="p">&lt;</span><span class="nc">term-list</span><span class="p">&gt;</span> &quot;)&quot;
<span class="p">&lt;</span><span class="nc">atom-list</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">atom</span><span class="p">&gt;</span> | <span class="p">&lt;</span><span class="nc">atom</span><span class="p">&gt;</span> &quot;,&quot; <span class="p">&lt;</span><span class="nc">atom-list</span><span class="p">&gt;</span> | &quot;&quot;
<span class="p">&lt;</span><span class="nc">term</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">constant</span><span class="p">&gt;</span> | <span class="p">&lt;</span><span class="nc">variable</span><span class="p">&gt;</span>
<span class="p">&lt;</span><span class="nc">term-list</span><span class="p">&gt;</span> <span class="o">::=</span> <span class="p">&lt;</span><span class="nc">term</span><span class="p">&gt;</span> | <span class="p">&lt;</span><span class="nc">term</span><span class="p">&gt;</span> &quot;,&quot; <span class="p">&lt;</span><span class="nc">term-list</span><span class="p">&gt;</span> | &quot;&quot;
</pre></div>
<p>Atoms are also referred to as <i><dfn>literals</dfn></i>. The atom to the left of the <code>:-</code> symbol is called the <i><dfn>head</dfn></i> of the rule; the atoms to the right are the <i><dfn>body</dfn></i>. Every Datalog program must satisfy the condition that every variable that appears in the head of a rule also appears in the body (this condition is sometimes called the <i><dfn>range restriction</dfn></i>).<sup id="cite_ref-FOOTNOTECeriGottlobTanca1989146_1-1" class="reference"><a href="#cite_note-FOOTNOTECeriGottlobTanca1989146-1">&#91;1&#93;</a></sup><sup id="cite_ref-2" class="reference"><a href="#cite_note-2">&#91;2&#93;</a></sup>
</p><p>There are two common conventions for variable names: capitalizing variables, or prefixing them with a question mark <code>?</code>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">&#91;3&#93;</a></sup>
</p><p>Note that under this definition, Datalog does <em>not</em> include negation nor aggregates; see <a href="#Extensions">§&#160;Extensions</a> for more information about those constructs.
</p><p>Rules with empty bodies are called <i><dfn>facts</dfn></i>. For example, the following rule is a fact:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">r</span><span class="p">(</span><span class="s s-Atom">x</span><span class="p">)</span> <span class="p">:-</span> <span class="p">.</span>
</pre></div>
<p>The set of facts is called the <i><dfn>extensional database</dfn></i> or <i><dfn>EDB</dfn></i> of the Datalog program. The set of tuples computed by evaluating the Datalog program is called the <i><dfn>intensional database</dfn></i> or <i><dfn>IDB</dfn></i>.
</p>
<h3><span class="mw-headline" id="Syntactic_sugar">Syntactic sugar</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=4" title="Edit section: Syntactic sugar"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Many implementations of logic programming extend the above grammar to allow writing facts without the <code>:-</code>, like so:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">r</span><span class="p">(</span><span class="s s-Atom">x</span><span class="p">).</span>
</pre></div>
<p>Some also allow writing 0-ary relations without parentheses, like so:
</p>
<div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">p</span> <span class="o">:-</span> <span class="s s-Atom">q</span><span class="p">.</span>
</pre></div>
<p>These are merely abbreviations (<a href="/wiki/Syntactic_sugar" title="Syntactic sugar">syntactic sugar</a>); they have no impact on the semantics of the program.
</p>
<h2><span class="mw-headline" id="Semantics">Semantics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=5" title="Edit section: Semantics"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>

<p>There are three widely-used approaches to the semantics of Datalog programs: <a href="/wiki/Model_theory" title="Model theory">model-theoretic</a>, <a href="/wiki/Fixed_point_(mathematics)" title="Fixed point (mathematics)">fixed-point</a>, and <a href="/wiki/Proof-theoretic_semantics" title="Proof-theoretic semantics">proof-theoretic</a>. These three approaches can be proven equivalent.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">&#91;4&#93;</a></sup>
</p><p>An atom is called <i><dfn><a href="/wiki/Ground_atom" class="mw-redirect" title="Ground atom">ground</a></dfn></i> if none of its subterms are variables. Intuitively, each of the semantics define the meaning of a program to be the set of all ground atoms that can be deduced from the rules of the program, starting from the facts.
</p>
<h3><span class="mw-headline" id="Model_theoretic">Model theoretic</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=6" title="Edit section: Model theoretic"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<p>A rule is called ground if all of its atoms (head and body) are ground. A ground rule <i>R</i><sub>1</sub> is a <i>ground instance</i> of another rule <i>R</i><sub>2</sub> if <i>R</i><sub>1</sub> is the result of a <a href="/wiki/Substitution_(logic)" title="Substitution (logic)">substitution</a> of constants for all the variables in <i>R</i><sub>2</sub>. The <i><a href="/wiki/Herbrand_base" class="mw-redirect" title="Herbrand base">Herbrand base</a></i> of a Datalog program is the set of all ground atoms that can be made with the constants appearing in the program. The <i><dfn>Herbrand model</dfn></i> of a Datalog program is the smallest subset of the Herbrand base such that, for each ground instance of each rule in the program, if the atoms in the body of the rule are in the set, then so is the head.<sup id="cite_ref-FOOTNOTECeriGottlobTanca1989149_5-0" class="reference"><a href="#cite_note-FOOTNOTECeriGottlobTanca1989149-5">&#91;5&#93;</a></sup> The model-theoretic semantics define the minimal Herbrand model to be the meaning of the program.
</p>
<h3><span class="mw-headline" id="Fixed-point">Fixed-point</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=7" title="Edit section: Fixed-point"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Let <span class="texhtml mvar" style="font-style:italic;">I</span> be the <a href="/wiki/Power_set" title="Power set">power set</a> of the Herbrand base of a program <i>P</i>. The <i>immediate consequence operator</i> for <i>P</i> is a map <span class="texhtml mvar" style="font-style:italic;">T</span> from <span class="texhtml mvar" style="font-style:italic;">I</span> to <span class="texhtml mvar" style="font-style:italic;">I</span> that adds all of the new ground atoms that can be derived from the rules of the program in a single step. The least-fixed-point semantics define the least fixed point of <span class="texhtml mvar" style="font-style:italic;">T</span> to be the meaning of the program; this coincides with the minimal Herbrand model.<sup id="cite_ref-FOOTNOTECeriGottlobTanca1989150_6-0" class="reference"><a href="#cite_note-FOOTNOTECeriGottlobTanca1989150-6">&#91;6&#93;</a></sup>
</p><p>The <a href="/wiki/Fixpoint" class="mw-redirect" title="Fixpoint">fixpoint</a> semantics suggest an algorithm for computing the minimal model: Start with the set of ground facts in the program, then repeatedly add consequences of the rules until a fixpoint is reached. This algorithm is called <a href="#Naïve_evaluation">naïve evaluation</a>.
</p>
<h3><span class="mw-headline" id="Proof-theoretic">Proof-theoretic</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=8" title="Edit section: Proof-theoretic"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Proof_tree_for_Datalog_transitive_closure_computation.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/0/04/Proof_tree_for_Datalog_transitive_closure_computation.svg/220px-Proof_tree_for_Datalog_transitive_closure_computation.svg.png" decoding="async" width="220" height="208" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/04/Proof_tree_for_Datalog_transitive_closure_computation.svg/330px-Proof_tree_for_Datalog_transitive_closure_computation.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/0/04/Proof_tree_for_Datalog_transitive_closure_computation.svg/440px-Proof_tree_for_Datalog_transitive_closure_computation.svg.png 2x" data-file-width="249" data-file-height="235" /></a><figcaption>Proof tree showing the derivation of the ground atom <code>path(x, z)</code> from the program  <div class="mw-highlight mw-highlight-lang-prolog mw-content-ltr" dir="ltr"><pre><span></span><span class="nf">edge</span><span class="p">(</span><span class="s s-Atom">x</span><span class="p">,</span> <span class="s s-Atom">y</span><span class="p">).</span>
<span class="nf">edge</span><span class="p">(</span><span class="s s-Atom">y</span><span class="p">,</span> <span class="s s-Atom">z</span><span class="p">).</span>

  <span class="nf">edge</span><span class="p">(</span><span class="nv">A</span><span class="p">,</span> <span class="nv">B</span><span class="p">).</span>

<p>The proof-theoretic semantics defines the meaning of a Datalog program to be the set of facts with corresponding <i>proof trees</i>. Intuitively, a proof tree shows how to derive a fact from the facts and rules of a program.
</p><p>One might be interested in knowing whether or not a particular ground atom appears in the minimal Herbrand model of a Datalog program, perhaps without caring much about the rest of the model. A top-down reading of the proof trees described above suggests an algorithm for computing the results of such <i>queries</i>. This reading informs the <a href="/wiki/SLD_resolution" title="SLD resolution">SLD resolution</a> algorithm, which forms the basis for the evaluation of <a href="/wiki/Prolog" title="Prolog">Prolog</a>.
</p>
<h2><span class="mw-headline" id="Evaluation">Evaluation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=9" title="Edit section: Evaluation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>There are many different ways to evaluate a Datalog program, with different performance characteristics.
</p>
<h3><span class="mw-headline" id="Bottom-up_evaluation_strategies">Bottom-up evaluation strategies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=10" title="Edit section: Bottom-up evaluation strategies"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Bottom-up evaluation strategies start with the facts in the program and repeatedly apply the rules until either some goal or query is established, or until the complete minimal model of the program is produced.
</p>
<h4><span id="Na.C3.AFve_evaluation"></span><span class="mw-headline" id="Naïve_evaluation">Naïve evaluation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=11" title="Edit section: Naïve evaluation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h4>
<p><i>Naïve evaluation</i> mirrors the <a href="#Fixpoint_Semantics">fixpoint semantics</a> for Datalog programs. Naïve evaluation uses a set of "known facts", which is initialized to the facts in the program. It proceeds by repeatedly enumerating all ground instances of each rule in the program. If each atom in the body of the ground instance is in the set of known facts, then the head atom is added to the set of known facts. This process is repeated until a fixed point is reached, and no more facts may be deduced. Naïve evaluation produces the entire minimal model of the program.<sup id="cite_ref-FOOTNOTECeriGottlobTanca1989154_7-0" class="reference"><a href="#cite_note-FOOTNOTECeriGottlobTanca1989154-7">&#91;7&#93;</a></sup>
</p>
<h4><span id="Semi-na.C3.AFve_evaluation"></span><span class="mw-headline" id="Semi-naïve_evaluation">Semi-naïve evaluation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=12" title="Edit section: Semi-naïve evaluation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h4>

<p>Semi-naïve evaluation is a bottom-up evaluation strategy that can be asymptotically faster than naïve evaluation.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">&#91;8&#93;</a></sup>
</p>
<h4><span class="mw-headline" id="Performance_considerations">Performance considerations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=13" title="Edit section: Performance considerations"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h4>
<figure class="mw-default-size" typeof="mw:File/Thumb"><a href="/wiki/File:Theta_supercomputer_-_389_071_002_(36954713450).jpg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/4/4d/Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg/220px-Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg" decoding="async" width="220" height="149" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/4/4d/Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg/330px-Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/4/4d/Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg/440px-Theta_supercomputer_-_389_071_002_%2836954713450%29.jpg 2x" data-file-width="2193" data-file-height="1490" /></a><figcaption>A parallel Datalog engine was evaluated on the Theta supercomputer at <a href="/wiki/Argonne_National_Laboratory" title="Argonne National Laboratory">Argonne National Laboratory</a>.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">&#91;9&#93;</a></sup></figcaption></figure>
<p>Naïve and semi-naïve evaluation both evaluate recursive Datalog rules by repeatedly applying them to a set of known facts until a fixed point is reached. In each iteration, rules are only run for "one step", i.e., non-recursively. As mentioned <a href="#Comparison_to_relational_databases">above</a>, each non-recursive Datalog rule corresponds precisely to a <a href="/wiki/Conjunctive_query" title="Conjunctive query">conjunctive query</a>. Therefore, many of the techniques from <a href="/wiki/Database_theory" title="Database theory">database theory</a> used to speed up conjunctive queries are applicable to bottom-up evaluation of Datalog, such as
</p>
<ul><li><a href="/wiki/Database_index" title="Database index">Index</a> selection<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">&#91;10&#93;</a></sup></li>
<li><a href="/wiki/Query_optimization" title="Query optimization">Query optimization</a>, especially join order<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">&#91;11&#93;</a></sup><sup id="cite_ref-12" class="reference"><a href="#cite_note-12">&#91;12&#93;</a></sup></li>
<li><a href="/wiki/Join_algorithm" class="mw-redirect" title="Join algorithm">Join algorithms</a></li>
<li>Selection of <a href="/wiki/Data_structure" title="Data structure">data structures</a> used to store relations; common choices include <a href="/wiki/Hash_table" title="Hash table">hash tables</a> and <a href="/wiki/B-tree" title="B-tree">B-trees</a>, other possibilities include <a href="/wiki/Disjoint-set_data_structure" title="Disjoint-set data structure">disjoint set data structures</a> (for storing <a href="/wiki/Equivalence_relation" title="Equivalence relation">equivalence relations</a>),<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">&#91;13&#93;</a></sup> bries (a variant of <a href="/wiki/Trie" title="Trie">tries</a>),<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">&#91;14&#93;</a></sup> <a href="/wiki/Binary_decision_diagram" title="Binary decision diagram">binary decision diagrams</a>,<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">&#91;15&#93;</a></sup> and even <a href="/wiki/Satisfiability_modulo_theories" title="Satisfiability modulo theories">SMT</a> formulas<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">&#91;16&#93;</a></sup></li></ul>
<p>Many such techniques are implemented in modern bottom-up Datalog engines such as <a href="/wiki/Souffl%C3%A9_(programming_language)" title="Soufflé (programming language)">Soufflé</a>. Some Datalog engines integrate SQL databases directly.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">&#91;17&#93;</a></sup>
</p><p>Bottom-up evaluation of Datalog is also amenable to <a href="/wiki/Parallel_computing" title="Parallel computing">parallelization</a>. Parallel Datalog engines are generally divided into two paradigms:
</p>
<ul><li>In the shared-memory, multi-core setting, Datalog engines execute on a single node. Coordination between threads may be achieved using <a href="/wiki/Lock_(computer_science)" title="Lock (computer science)">locking</a> or <a href="/wiki/Lock-free_data_structures" class="mw-redirect" title="Lock-free data structures">lock-free data structures</a>. The shared-memory setting may be further divided into <a href="/wiki/Single_instruction,_multiple_data" title="Single instruction, multiple data">single instruction, multiple data</a> and <a href="/wiki/Multiple_instruction,_multiple_data" title="Multiple instruction, multiple data">multiple instruction, multiple data</a> paradigms:
<ul><li>Datalog engines that execute on <a href="/wiki/Graphics_processing_unit" title="Graphics processing unit">graphics processing units</a> fall into the SIMD paradigm.<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">&#91;18&#93;</a></sup></li>
<li>Datalog engines using <a href="/wiki/OpenMP" title="OpenMP">OpenMP</a><sup id="cite_ref-19" class="reference"><a href="#cite_note-19">&#91;19&#93;</a></sup> are instances of the MIMD paradigm.</li></ul></li>
<li>In the <a href="/wiki/Shared-nothing_architecture" title="Shared-nothing architecture">shared-nothing</a> setting, Datalog engines execute on a <a href="/wiki/Computer_cluster" title="Computer cluster">cluster</a> of nodes. Such engines generally operate by splitting relations into disjoint subsets based on a <a href="/wiki/Hash_function" title="Hash function">hash function</a>, performing computations (joins) on each node, and then exchanging newly-generated tuples over the network.<sup id="cite_ref-20" class="reference"><a href="#cite_note-20">&#91;20&#93;</a></sup> Examples include Datalog engines based on <a href="/wiki/Message_Passing_Interface" title="Message Passing Interface">MPI</a>,<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">&#91;21&#93;</a></sup> <a href="/wiki/Apache_Hadoop" title="Apache Hadoop">Hadoop</a>,<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">&#91;22&#93;</a></sup> and <a href="/wiki/Apache_Spark" title="Apache Spark">Spark</a>.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">&#91;23&#93;</a></sup></li></ul>
<h3><span class="mw-headline" id="Top-down_evaluation_strategies">Top-down evaluation strategies</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=14" title="Edit section: Top-down evaluation strategies"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1097763485"><table class="box-Expand_section plainlinks metadata ambox mbox-small-left ambox-content" role="presentation"><tbody><tr><td class="mbox-image"><span typeof="mw:File"><a href="/wiki/File:Wiki_letter_w_cropped.svg" class="mw-file-description"><img alt="[icon]" src="//upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/20px-Wiki_letter_w_cropped.svg.png" decoding="async" width="20" height="14" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/30px-Wiki_letter_w_cropped.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/40px-Wiki_letter_w_cropped.svg.png 2x" data-file-width="44" data-file-height="31" /></a></span></td><td class="mbox-text"><div class="mbox-text-span">This section <b>needs expansion</b>. You can help by <a class="external text" href="https://en.wikipedia.org/w/index.php?title=Datalog&amp;action=edit&amp;section=">adding to it</a>.  <span class="date-container"><i>(<span class="date">March 2023</span>)</i></span></div></td></tr></tbody></table>
<p><a href="/wiki/SLD_resolution" title="SLD resolution">SLD resolution</a> is sound and complete for Datalog programs.
</p>
<h3><span class="mw-headline" id="Magic_sets">Magic sets</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=15" title="Edit section: Magic sets"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<p>Top-down evaluation strategies begin with a <i>query</i> or <i>goal</i>. Bottom-up evaluation strategies can answer queries by computing the entire minimal model and matching the query against it, but this can be inefficient if the answer only depends on a small subset of the entire model. The <i>magic sets</i> algorithm takes a Datalog program and a query, and produces a more efficient program that computes the same answer to the query while still using bottom-up evaluation.<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">&#91;24&#93;</a></sup> A variant of the magic sets algorithm has been shown to produce programs that, when evaluated using <a href="#Semi-naïve_evaluation">semi-naïve evaluation</a>, are as efficient as top-down evaluation.<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">&#91;25&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="Complexity">Complexity</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=16" title="Edit section: Complexity"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <a href="/wiki/Decision_problem" title="Decision problem">decision problem</a> formulation of Datalog evaluation is as follows: Given a Datalog program <span class="texhtml mvar" style="font-style:italic;">P</span> split into a set of facts (EDB) <span class="texhtml mvar" style="font-style:italic;">E</span> and a set of rules <span class="texhtml mvar" style="font-style:italic;">R</span>, and a ground atom <span class="texhtml mvar" style="font-style:italic;">A</span>, is <span class="texhtml mvar" style="font-style:italic;">A</span> in the minimal model of <span class="texhtml mvar" style="font-style:italic;">P</span>? In this formulation, there are three variations of the <a href="/wiki/Computational_complexity" title="Computational complexity">computational complexity</a> of evaluating Datalog programs:<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">&#91;26&#93;</a></sup>
</p>
<ul><li>The <i><dfn>data complexity</dfn></i> is the complexity of the decision problem when <span class="texhtml mvar" style="font-style:italic;">A</span> and <span class="texhtml mvar" style="font-style:italic;">E</span> are inputs and <span class="texhtml mvar" style="font-style:italic;">R</span> is fixed.</li>
<li>The <i><dfn>program complexity</dfn></i> is the complexity of the decision problem when <span class="texhtml mvar" style="font-style:italic;">A</span> and <span class="texhtml mvar" style="font-style:italic;">R</span> are inputs and <span class="texhtml mvar" style="font-style:italic;">E</span> is fixed.</li>
<li>The <i><dfn>combined complexity</dfn></i> is the complexity of the decision problem when <span class="texhtml mvar" style="font-style:italic;">A</span>, <span class="texhtml mvar" style="font-style:italic;">E</span>, and <span class="texhtml mvar" style="font-style:italic;">R</span> are inputs.</li></ul>
<p>With respect to data complexity, the decision problem for Datalog is <a href="/wiki/P-complete" title="P-complete">P-complete</a>. With respect to program complexity, the decision problem is <a href="/wiki/EXPTIME-complete" class="mw-redirect" title="EXPTIME-complete">EXPTIME-complete</a>. In particular, evaluating Datalog programs always terminates; Datalog is not <a href="/wiki/Turing_completeness" title="Turing completeness">Turing-complete</a>.
</p><p>Some extensions to Datalog do not preserve these complexity bounds. Extensions implemented in some <a href="#Datalog_engines">Datalog engines</a>, such as algebraic data types, can even make the resulting language Turing-complete.
</p>
<h2><span class="mw-headline" id="Extensions">Extensions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=17" title="Edit section: Extensions"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Several extensions have been made to Datalog, e.g., to support negation, <a href="/wiki/Aggregate_function" title="Aggregate function">aggregate functions</a>, inequalities, to allow <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>, or to allow <a href="/wiki/Disjunction" class="mw-redirect" title="Disjunction">disjunctions</a> as heads of <a href="/wiki/Clause_(logic)" title="Clause (logic)">clauses</a>. These extensions have significant impacts on the language's semantics and on the implementation of a corresponding interpreter.
</p><p>Datalog is a syntactic subset of <a href="/wiki/Prolog" title="Prolog">Prolog</a>, <a href="/wiki/Disjunctive_Datalog" title="Disjunctive Datalog">disjunctive Datalog</a>, <a href="/wiki/Answer_set_programming" title="Answer set programming">answer set programming</a>, <a href="/wiki/DatalogZ" title="DatalogZ">DatalogZ</a>, and <a href="/wiki/Constraint_logic_programming" title="Constraint logic programming">constraint logic programming</a>. When evaluated as an answer set program, a Datalog program yields a single answer set, which is exactly its minimal model.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">&#91;27&#93;</a></sup>
</p><p>Many implementations of Datalog extend Datalog with additional features; see <a href="#Datalog_engines">§&#160;Datalog engines</a> for more information.
</p>
<h3><span class="mw-headline" id="Aggregation">Aggregation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=18" title="Edit section: Aggregation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1097763485"><table class="box-Expand_section plainlinks metadata ambox mbox-small-left ambox-content" role="presentation"><tbody><tr><td class="mbox-image"><span typeof="mw:File"><a href="/wiki/File:Wiki_letter_w_cropped.svg" class="mw-file-description"><img alt="[icon]" src="//upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/20px-Wiki_letter_w_cropped.svg.png" decoding="async" width="20" height="14" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/30px-Wiki_letter_w_cropped.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/1/1c/Wiki_letter_w_cropped.svg/40px-Wiki_letter_w_cropped.svg.png 2x" data-file-width="44" data-file-height="31" /></a></span></td><td class="mbox-text"><div class="mbox-text-span">This section <b>needs expansion</b>. <p>Datalog can be extended to support <a href="/wiki/Aggregate_function" title="Aggregate function">aggregate functions</a>.<sup id="cite_ref-28" class="reference"><a href="#cite_note-28">&#91;28&#93;</a></sup>
</p><p>Notable Datalog engines that implement aggregation include:
</p>
<ul><li><a href="/wiki/LogicBlox" title="LogicBlox">LogicBlox</a><sup id="cite_ref-29" class="reference"><a href="#cite_note-29">&#91;29&#93;</a></sup></li>
<li><a href="/wiki/Souffl%C3%A9_(programming_language)" title="Soufflé (programming language)">Soufflé</a></li></ul>
<h3><span class="mw-headline" id="Negation">Negation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=19" title="Edit section: Negation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h3>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1033289096"><div role="note" class="hatnote navigation-not-searchable">Further information: <a href="/wiki/Syntax_and_semantics_of_logic_programming#Extending_Datalog_with_negation" title="Syntax and semantics of logic programming">Syntax and semantics of logic programming §&#160;Extending Datalog with negation</a></div>
<p>Adding negation to Datalog complicates its semantics, leading to whole new languages and strategies for evaluation. For example, the language that results from adding negation with the <a href="/wiki/Stable_model_semantics" title="Stable model semantics">stable model semantics</a> is exactly <a href="/wiki/Answer_set_programming" title="Answer set programming">answer set programming</a>.
</p><p><a href="/wiki/Stratification_(mathematics)#In_mathematical_logic" title="Stratification (mathematics)">Stratified</a> negation can be added to Datalog while retaining its model-theoretic and fixed-point semantics. Notable Datalog engines that implement stratified negation include:
</p>
<ul><li><a href="/wiki/LogicBlox" title="LogicBlox">LogicBlox</a><sup id="cite_ref-30" class="reference"><a href="#cite_note-30">&#91;30&#93;</a></sup></li>
<li><a href="/wiki/Souffl%C3%A9_(programming_language)" title="Soufflé (programming language)">Soufflé</a></li></ul>
<h2><span class="mw-headline" id="Comparison_to_Prolog">Comparison to Prolog</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=20" title="Edit section: Comparison to Prolog"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Unlike in <a href="/wiki/Prolog" title="Prolog">Prolog</a>, statements of a Datalog program can be stated in any order. Datalog does not have Prolog's <a href="/wiki/Cut_(logic_programming)" title="Cut (logic programming)">cut</a> operator. This makes Datalog a fully <a href="/wiki/Declarative_language" class="mw-redirect" title="Declarative language">declarative language</a>.
</p><p>In contrast to Prolog, Datalog
</p>
<ul><li>disallows complex terms as arguments of <a href="/wiki/Predicate_(logic)" class="mw-redirect" title="Predicate (logic)">predicates</a>, e.g., <code>p(x, y)</code> is admissible but not <code>p(f(x), y)</code>,</li>
<li>disallows negation,</li>
<li>requires that every variable that appears in the head of a <a href="/wiki/Clause_(logic)" title="Clause (logic)">clause</a> also appear in a <a href="/wiki/Literal_(mathematical_logic)" title="Literal (mathematical logic)">literal</a> in the body of the clause.</li></ul>
<p>This article deals primarily with Datalog without negation (see also <a href="/wiki/Syntax_and_semantics_of_logic_programming#Extending_Datalog_with_negation" title="Syntax and semantics of logic programming">Syntax and semantics of logic programming §&#160;Extending Datalog with negation</a>). However, stratified negation is a common addition to Datalog; the following list contrasts <a href="/wiki/Prolog" title="Prolog">Prolog</a> with Datalog with stratified negation. Datalog with stratified negation
</p>
<ul><li>also disallows complex terms as arguments of <a href="/wiki/Predicate_(logic)" class="mw-redirect" title="Predicate (logic)">predicates</a>,</li>
<li>requires that every variable that appears in the head of a <a href="/wiki/Clause_(logic)" title="Clause (logic)">clause</a> also appear in a positive (i.e., not negated) atom in the body of the clause,</li>
<li>requires that every variable appearing in a negative literal in the body of a clause also appear in some positive literal in the body of the clause.<sup id="cite_ref-31" class="reference"><a href="#cite_note-31">&#91;31&#93;</a></sup><sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Reliable_sources" title="Wikipedia:Reliable sources"><span title="The material near this tag may rely on an unreliable source. (March 2023)">unreliable source?</span></a></i>&#93;</sup></li></ul>
<h2><span class="mw-headline" id="Expressiveness">Expressiveness</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=21" title="Edit section: Expressiveness"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The <i><dfn>boundedness problem</dfn></i> for Datalog asks, given a Datalog program, whether it is <i><dfn>bounded</dfn></i>, i.e., the maximal recursion depth reached when evaluating the program on an input database can be bounded by some constant. In other words, this question asks whether the Datalog program could be rewritten as a nonrecursive Datalog program. Solving the boundedness problem on arbitrary Datalog programs is <a href="/wiki/Undecidable_problem" title="Undecidable problem">undecidable</a>,<sup id="cite_ref-32" class="reference"><a href="#cite_note-32">&#91;32&#93;</a></sup> but it can be made decidable by restricting to some fragments of Datalog.
</p>
<h2><span class="mw-headline" id="Datalog_engines">Datalog engines</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=22" title="Edit section: Datalog engines"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Systems that implement languages inspired by Datalog, whether <a href="/wiki/Compiler" title="Compiler">compilers</a>, <a href="/wiki/Interpreter_(computing)" title="Interpreter (computing)">interpreters</a>, <a href="/wiki/Library_(computing)" title="Library (computing)">libraries</a>, or <a href="/wiki/Embedded_DSL" class="mw-redirect" title="Embedded DSL">embedded DSLs</a>, are referred to as <i><dfn>Datalog engines</dfn></i>. Datalog engines often implement extensions of Datalog, extending it with additional <a href="/wiki/Data_type" title="Data type">data types</a>, <a href="/wiki/Foreign_function_interface" title="Foreign function interface">foreign function interfaces</a>, or support for user-defined <a href="/wiki/Lattice_(order)" title="Lattice (order)">lattices</a>. Such extensions may allow for writing <a href="/wiki/Divergence_(computer_science)" title="Divergence (computer science)">non-terminating</a> or otherwise ill-defined programs.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (October 2023)">citation needed</span></a></i>&#93;</sup>
</p>
<h2><span class="mw-headline" id="Uses_and_influence">Uses and influence</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=23" title="Edit section: Uses and influence"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>Datalog is quite limited in its expressivity. It is not <a href="/wiki/Turing_completeness" title="Turing completeness">Turing-complete</a>, and doesn't include basic data types such as <a href="/wiki/Integer_(computer_science)" title="Integer (computer science)">integers</a> or <a href="/wiki/String_(computer_science)" title="String (computer science)">strings</a>. This parsimony is appealing from a theoretical standpoint, but it means Datalog <i>per se</i> is rarely used as a programming language or <a href="/wiki/Knowledge_representation" class="mw-redirect" title="Knowledge representation">knowledge representation</a> language.<sup id="cite_ref-33" class="reference"><a href="#cite_note-33">&#91;33&#93;</a></sup> Most <a href="#Datalog_engines">Datalog engines</a> implement substantial extensions of Datalog. However, Datalog has a strong influence on such implementations, and many authors don't bother to distinguish them from Datalog as presented in this article. Accordingly, the applications discussed in this section include applications of realistic implementations of Datalog-based languages.
</p><p>Datalog has been applied to problems in <a href="/wiki/Data_integration" title="Data integration">data integration</a>, <a href="/wiki/Information_extraction" title="Information extraction">information extraction</a>, <a href="/wiki/Computer_network" title="Computer network">networking</a>, <a href="/wiki/Security" title="Security">security</a>, <a href="/wiki/Cloud_computing" title="Cloud computing">cloud computing</a> and <a href="/wiki/Machine_learning" title="Machine learning">machine learning</a>.<sup id="cite_ref-34" class="reference"><a href="#cite_note-34">&#91;34&#93;</a></sup><sup id="cite_ref-35" class="reference"><a href="#cite_note-35">&#91;35&#93;</a></sup> <a href="/wiki/Google" title="Google">Google</a> has developed an extension to Datalog for <a href="/wiki/Big_data" title="Big data">big data</a> processing.<sup id="cite_ref-36" class="reference"><a href="#cite_note-36">&#91;36&#93;</a></sup>
</p><p>Datalog has seen application in <a href="/wiki/Static_program_analysis" title="Static program analysis">static program analysis</a>.<sup id="cite_ref-37" class="reference"><a href="#cite_note-37">&#91;37&#93;</a></sup> The <a href="/wiki/Souffl%C3%A9_(programming_language)" title="Soufflé (programming language)">Soufflé</a> dialect has been used to write <a href="/wiki/Pointer_analysis" title="Pointer analysis">pointer analyses</a> for <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> and a <a href="/wiki/Control_flow_analysis" title="Control flow analysis">control-flow analysis</a> for <a href="/wiki/Scheme_(programming_language)" title="Scheme (programming language)">Scheme</a>.<sup id="cite_ref-38" class="reference"><a href="#cite_note-38">&#91;38&#93;</a></sup><sup id="cite_ref-39" class="reference"><a href="#cite_note-39">&#91;39&#93;</a></sup> Datalog has been integrated with <a href="/wiki/Satisfiability_modulo_theories" title="Satisfiability modulo theories">SMT solvers</a> to make it easier to write certain static analyses.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40">&#91;40&#93;</a></sup> The <a href="/wiki/Flix_(programming_language)" title="Flix (programming language)">Flix</a> dialect is also suited to writing static program analyses.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">&#91;41&#93;</a></sup>
</p><p>Some widely used database systems include ideas and algorithms developed for Datalog. For example, the <a href="/wiki/SQL:1999" title="SQL:1999">SQL:1999</a> standard includes <a href="/wiki/Hierarchical_and_recursive_queries_in_SQL" title="Hierarchical and recursive queries in SQL">recursive queries</a>, and the Magic Sets algorithm (initially developed for the faster evaluation of Datalog queries) is implemented in IBM's <a href="/wiki/IBM_DB2" class="mw-redirect" title="IBM DB2">DB2</a>.<sup id="cite_ref-42" class="reference"><a href="#cite_note-42">&#91;42&#93;</a></sup>
</p>
<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Datalog&amp;action=edit&amp;section=24" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></h2>
<p>The origins of Datalog date back to the beginning of <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a>, but it became prominent as a separate area around 1977 when Hervé Gallaire and <a href="/wiki/Jack_Minker" title="Jack Minker">Jack Minker</a> organized a workshop on <a href="/wiki/Logic" title="Logic">logic</a> and <a href="/wiki/Database" title="Database">databases</a>.<sup id="cite_ref-43" class="reference"><a href="#cite_note-43">&#91;43&#93;</a></sup> <a href="/wiki/David_Maier" title="David Maier">David Maier</a> is credited with coining the term Datalog.<sup id="cite_ref-44" class="reference"><a href="#cite_note-44">&#91;44&#93;</a></sup>
</p>
