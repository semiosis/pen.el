<article class="unison-doc"><section><table><tbody><tr><td><strong><span class="span"><span class="word">The problem</span></span></strong></td><td><strong><span class="span"><span class="word">Our remedy</span></span></strong></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Cloud infrastructure management overhead:</span></span></strong><span class="word"> </span><span class="word">Managing cloud infrastructure is complicated and not all teams have the expertise or desire to do it themselves. Furthermore, managing cloud infrastructure isn't done with a nicely typed program; instead, it's lots of ad hoc tooling, shell scripts and YAML files.</span></span></td><td><span class="span"><strong><span class="span"><span class="word">Unison code deploys itself.</span></span></strong><span class="word"> </span><a class="named-link" href="https://www.unison.cloud/" rel="noopener" target="_blank"><span class="span"><span class="word">Our cloud computing platform, Unison Cloud,</span></span></a><span class="word"> </span><span class="word">makes deploying long-running services, serverless functions, and batch jobs to the cloud as easy as calling a regular function. Infrastructure management is written as a typed Unison program.</span></span></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Inter-service communication boilerplate:</span></span></strong><span class="word"> </span><span class="word">Writing encoders and decoders at service boundaries is part of the tedious overhead we've accepted for microservice architectures.</span></span></td><td><span class="span"><strong><span class="span"><span class="word">Native service calls</span></span></strong><span class="word"> </span><span class="word">in Unison eliminate the need for intermediaries.</span><span class="word"> </span><a class="named-link" href="https://www.unison-lang.org/whats-new/unison-services-preview/" rel="noopener" target="_blank"><span class="span"><span class="word">Service calls are typechecked function calls.</span></span></a></span></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Database communication boilerplate:</span></span></strong><span class="word"> </span><span class="word">Translating your typed domain models to something the database understands is tedious and loses type information.</span></span></td><td><span class="span"><strong><span class="span"><span class="word">Unison supports native, typed, transactional storage in the Cloud.</span></span></strong><span class="word"> </span><span class="word">That means no writing sql adapters, no special query languages, just store and access Unison data.</span></span></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Testing and profiling distributed systems:</span></span></strong><span class="word"> </span><span class="word">Testing the integration of an entire distributed system often involves setup and teardown of resources or copious data mocking.</span></span></td><td><span class="span"><strong><span class="span"><span class="word">Unison's effect system separates your business logic from its implementation.</span></span></strong><span class="word"> </span><a class="named-link" href="https://share.unison-lang.org/@unison/cloud" rel="noopener" target="_blank"><span class="span"><span class="word">Our model for distributed computation</span></span></a><span class="word"> </span><span class="word">can run against test backends that focus on observability, tracing or performance.</span></span></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Unnecessary merge conflicts:</span></span></strong><span class="word"> </span><span class="word">Dealing with semantically unimportant merge conflicts is at best a papercut and at worst a big waste of development time.</span></span></td><td><span class="span"><strong><span class="span"><span class="word">Semantic version control.</span></span></strong><span class="word"> </span><span class="word">Unison's version control is language-aware. There are no conflicts due to code formatting or whitespace, order of imports, order of definitions in a file, or any other differences that aren't semantically meaningful.</span></span></td></tr><tr><td><span class="span"><strong><span class="span"><span class="word">Dependency management nightmares:</span></span></strong><span class="word"> </span><span class="word">Finding compatible sets of library versions can be difficult or impossible due to dependency conflicts.
