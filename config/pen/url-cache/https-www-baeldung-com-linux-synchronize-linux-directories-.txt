<section class="post-content clearfix" itemprop="articleBody">
<h2 id="bd-overview" data-id="overview">1. Overview</h2>
<div class="bd-anchor" id="overview"></div>
<p>Syncronizing Linux directories in both directions ensures changes made in one directory are mirrored or updated in another, and vice versa.</p>
<p>In this tutorial, we&#8217;ll test several commands or programs to synchronize directories in both directions on a single machine, and then compare the results.</p>
<p>All commands mentioned in this guide have been tested on 64-bit Debian 10.10 (Buster), running GNU Bash 5.1.4, <a href="https://man7.org/linux/man-pages/man1/rsync.1.html"><em>rsync</em></a> 3.1.3, 64-bit <a href="https://www.cis.upenn.edu/~bcpierce/unison/">Unison</a> 2.51.4 with OCaml 4.05.0 binary, <a href="https://github.com/Fitus/Zaloha2.sh/tree/37c8ede62020213d12edf0747d3cc4a1a3f70875">Zaloha2</a> Bash script, and <a href="https://freefilesync.org/">FreeFileSync</a> 11.16.</p>
<h2 id="bd-using-the-rsync-command" data-id="using-the-rsync-command">2. Using the <em>rsync</em> Command</h2>
<div class="bd-anchor" id="using-the-rsync-command"></div>
<h3 id="bd-1-installation" data-id="1-installation">2.1. Installation</h3>
<div class="bd-anchor" id="1-installation"></div>
<p><em>rsync</em> is commonly found on Unix-like operating systems. Its source code or binary can also be downloaded from its <a href="https://rsync.samba.org/">homepage</a>.</p>
<h3 id="bd-2-usage" data-id="2-usage">2.2. Usage</h3>
<div class="bd-anchor" id="2-usage"></div>
<p><em>rsync</em> works in one direction, so we need to <strong>run it twice to sync directories in both directions</strong>.</p><div class="code-block code-block-2" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__336x280 __728x90" id="baeldung_leaderboard_mid_1">
<script data-cfasync="false" type="text/javascript">
    freestar.config.enabled_slots.push({ placementName: "baeldung_leaderboard_mid_1", slotId: "baeldung_leaderboard_mid_1" });
  </script>
</div></div>
<p>To illustrate the sync operation, let&#8217;s first create a sample directory structure:</p>
<pre><code class="language-bash">$ mkdir source target
$ touch source/file01.bin source/file02.bin source/file03.bin
$ touch target/file0A.bin target/file0B.bin target/file0C.bin

$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin

└── target
    ├── file0A.bin
    ├── file0B.bin
    └── file0C.bin

2 directories, 6 files</code></pre>
<p>Now that we have a sample directory, let&#8217;s sync from source to target:</p>
<pre><code class="language-bash">$ rsync -Pcauv source/ target/
sending incremental file list
./
file01.bin
              0 100%    0.00kB/s    0:00:00 (xfr#1, to-chk=2/4)
file02.bin
              0 100%    0.00kB/s    0:00:00 (xfr#2, to-chk=1/4)
file03.bin

total size is 0  speedup is 0.00

$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   └── file03.bin
└── target
    ├── file01.bin
    ├── file02.bin
    ├── file03.bin
    ├── file0A.bin
    ├── file0B.bin
    └── file0C.bin

2 directories, 9 files</code></pre>
<p>Then, let&#8217;s sync from target to source:</p>
<pre><code class="language-bash">$ rsync -Pcauv target/ source/
sending incremental file list
file0A.bin
              0 100%    0.00kB/s    0:00:00 (xfr#1, to-chk=2/7)
file0B.bin
              0 100%    0.00kB/s    0:00:00 (xfr#2, to-chk=1/7)
file0C.bin
              0 100%    0.00kB/s    0:00:00 (xfr#3, to-chk=0/7)

sent 401 bytes  received 73 bytes  948.00 bytes/sec
total size is 0  speedup is 0.00

$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   ├── file03.bin
│   ├── file0A.bin
│   ├── file0B.bin
│   └── file0C.bin
└── target
    ├── file01.bin
    ├── file02.bin
    ├── file03.bin
    ├── file0A.bin
    ├── file0B.bin
    └── file0C.bin

2 directories, 12 files</code></pre>
<p>As a result of the <em>rsync</em> command that we ran twice in both directions, both directories are now identical.</p>
<p>Let&#8217;s review the options that we used:</p><div class="code-block code-block-3" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__300x250 __336x280" id="baeldung_leaderboard_mid_2">
<script data-cfasync="false" type="text/javascript">
    freestar.config.enabled_slots.push({ placementName: "baeldung_leaderboard_mid_2", slotId: "baeldung_leaderboard_mid_2" });
  </script>
</div></div>
<ul>
<li><em>-P</em>: display statistics of the whole transfer rather than individual files</li>
<li><em>-c</em>: skip object based on checksum, not on modified time and size</li>
<li><em>-a</em>: enable archive mode, equals to:
<ul>
<li><em>-r</em>: recurse into directories</li>
<li><em>-l</em>: copy symlinks as symlinks</li>
<li><em>-p</em>: preserve permissions</li>
<li><em>-t</em>: preserve modification times</li>
<li><em>-g</em>: preserve group</li>
<li><em>-o</em>: preserve owner</li>
<li><em>-D</em>: preserve device files and special files</li>
</ul>
</li>
<li><em>-u</em>: skip newer files on the target</li>
<li><em>-v</em>: display detailed information of what is being done</li>
</ul>
<p>Additionally, we can also do a <strong>dry run by passing the <em>-n</em> option</strong>, which will <strong>simulate the sync with no changes made</strong>:</p>
<pre><code class="language-bash">$ rsync -Pcauvn source/ target/
sending incremental file list
./
file01.bin
file02.bin
file03.bin

sent 189 bytes  received 28 bytes  434.00 bytes/sec
total size is 0  speedup is 0.00 (DRY RUN)</code></pre>
<p>The output in the last line, <em>(DRY RUN)</em>, indicates that it&#8217;s a dry run. This is particularly useful if we need to <strong>verify that the command will succeed without any errors</strong>, such as incorrect options, or insufficient disk space, or any other potential issues.</p>
<h2 id="bd-using-unison-program" data-id="using-unison-program">3. Using Unison Program</h2>
<div class="bd-anchor" id="using-unison-program"></div>
<h3 id="bd-1-installation-1" data-id="1-installation-1">3.1. Installation</h3>
<div class="bd-anchor" id="1-installation-1"></div>
<p>Unison has its own <a href="https://www.cis.upenn.edu/~bcpierce/unison/">homepage</a>, however, it hosts the source code and binaries on a GitHub <a href="https://github.com/bcpierce00/unison">repository</a>.</p>
<p>We&#8217;re using <a href="https://github.com/bcpierce00/unison/releases/tag/v2.51.4">64-bit Unison 2.51.4 with OCaml 4.05.0 binary</a>, which gives us this package:</p>
<pre><code class="language-bash">$ tree
.
├── bin
│   ├── unison
│   ├── unison-fsmonitor
│   └── unison-gtk2
├── LICENSE
└── README.md

1 directory, 5 files</code></pre>
<p>The package provides 3 executable files: text-based (<em>unison</em>), GUI-based (<em>unison-gtk2</em>), and file system monitoring (<em>unison-fsmonitor</em>). However, for this article, we&#8217;ll use the text-based Unison.</p><div class="code-block code-block-4" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__300x250 __336x280" id="baeldung_leaderboard_mid_3">
<script data-cfasync="false" type="text/javascript">

<h3 id="bd-2-usage-1" data-id="2-usage-1">3.2. Usage</h3>
<div class="bd-anchor" id="2-usage-1"></div>
<p>To demonstrate its usage, let&#8217;s say we have the following structure of <em>source</em> and <em>target</em> directories:</p>
<pre><code class="language-bash">$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   └── file03.bin
└── target
    ├── file0A.bin
    ├── file0B.bin
    └── file0C.bin

2 directories, 6 files</code></pre>
<p>Whenever we sync a new source and target directory for the first time, which Unison refers to as a profile, we&#8217;ll get this interactive prompt:</p>
<pre><code class="language-bash">$ ./unison ~/source/ ~/target/
Unison 2.51.4 (ocaml 4.05.0): Contacting server...
Looking for changes
Warning: No archive files were found for these roots, whose canonical names are:
	~/source
	~/target
This can happen either because this is the first time you have synchronized these roots, or because you have upgraded Unison to a new version with a different archive format.  
...
Donations to the Unison project are gratefully accepted: http://www.cis.upenn.edu/~bcpierce/unison

Press return to continue.[&lt;spc&gt;] </code></pre>
<p>We can press <em>Enter</em> to continue, or <em>q</em> to quit. If we press <em>Enter</em>, it will ask for confirmation for each change that we want to do.</p>
<p>We press <em>f</em> or <em>Enter</em> to confirm each change. We can press <em>&gt;</em> or <em>&lt;</em> to force the change to be propagated from left to right, or right to left, or press <em>/</em> to skip the file and leave both objects alone.</p>
<p>When we reach the end of the list, Unison will ask one more time if we want to proceed with the changes, and we can press <em>y</em> to proceed, or <em>q</em> to terminate.</p>
<p>Every time Unison asks for input, we can press <em>?</em> to list all possible responses and their meanings:</p>
<pre><code class="language-bash">Reconciling changes

source         target             
file     ----&gt;            file01.bin  [f] f
file     ----&gt;            file02.bin  [f] f
file     ----&gt;            file03.bin  [f] f
         &lt;---- file       file0A.bin  [f] f
         &lt;---- file       file0B.bin  [f] f
         &lt;---- file       file0C.bin  [f] f

Proceed with propagating updates? [] y
Propagating updates

Unison 2.51.4 (ocaml 4.05.0) started propagating changes at 15:29:09.85 on 06 Jan 2022
[BGN] Copying file01.bin from ~/source to ~/target
[END] Copying file01.bin
[BGN] Copying file02.bin from ~/source to ~/target
[END] Copying file02.bin
[BGN] Copying file03.bin from ~/source to ~/target
[END] Copying file03.bin
[BGN] Copying file0A.bin from ~/target to ~/source
[END] Copying file0A.bin
[BGN] Copying file0B.bin from ~/target to ~/source
[END] Copying file0B.bin
[BGN] Copying file0C.bin from ~/target to ~/source
[END] Copying file0C.bin
Unison 2.51.4 (ocaml 4.05.0) finished propagating changes at 15:29:09.86 on 06 Jan 2022, 0.012 s

Saving synchronizer state
Synchronization complete at 15:29:09  (6 items transferred, 0 skipped, 0 failed)</code></pre>
<p>Once it&#8217;s done, if we try to sync again, it&#8217;ll say nothing to sync:</p>
<pre><code class="language-bash">$ ./unison ~/source/ ~/target/
Unison 2.51.4 (ocaml 4.05.0): Contacting server...
Looking for changes
Reconciling changes
Nothing to do: replicas have not changed since last sync.</code></pre>
<p>Let&#8217;s delete and modify some files, then sync again:</p><div class="code-block code-block-5" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__336x280 __336x280" id="baeldung_incontent_1">
<script data-cfasync="false" type="text/javascript">
    freestar.config.enabled_slots.push({ placementName: "baeldung_incontent_1", slotId: "baeldung_incontent_1" });
  </script>
</div></div>
<pre><code class="language-bash">$ rm source/file0B.bin 
$ echo &quot;edit file&quot; &gt; source/file0C.bin 
$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   ├── file03.bin
│   ├── file0A.bin

    ├── file02.bin
    ├── file03.bin
    ├── file0A.bin
    ├── file0B.bin
    └── file0C.bin

2 directories, 11 files

<p>Both directories are now identical.</p>
<h3 id="bd-3-non-interactive-mode" data-id="3-non-interactive-mode">3.3. Non-interactive Mode</h3>
<div class="bd-anchor" id="3-non-interactive-mode"></div>
<p>If we want to <strong>immediately proceed to sync with default Unison settings without any interactive prompt</strong>, where <strong>non-conflicting changes will be propagated and conflicts will be skipped</strong>, we can pass <em>-batch=true</em> as an option:</p>
<pre><code class="language-bash">$ ./unison -batch=true ~/source/ ~/target/
Unison 2.51.4 (ocaml 4.05.0): Contacting server...
Looking for changes
Reconciling changes
changed  ----&gt;            file0C.bin  
source       : changed file       modified on 2022-01-06 at 16:40:13  size 11        rw-r--r--
target       : unchanged file     modified on 2022-01-06 at 16:28:13  size 11        rw-r--r--
Propagating updates
Unison 2.51.4 (ocaml 4.05.0) started propagating changes at 16:40:35.93 on 06 Jan 2022
[BGN] Updating file file0C.bin from ~/source to ~/target
[END] Updating file file0C.bin
Unison 2.51.4 (ocaml 4.05.0) finished propagating changes at 16:40:35.93 on 06 Jan 2022, 0.000 s
Saving synchronizer state
Synchronization complete at 16:40:35  (1 item transferred, 0 skipped, 0 failed)</code></pre>
<p>By using the <em>batch</em> option, we can <strong><a href="/linux/cron-time-ranges">create a script or a cron job</a> to do the sync automatically and periodically</strong>.</p>
<h2 id="bd-using-zaloha2-bash-script" data-id="using-zaloha2-bash-script">4. Using Zaloha2 Bash Script</h2>
<div class="bd-anchor" id="using-zaloha2-bash-script"></div>
<h3 id="bd-1-installation-2" data-id="1-installation-2">4.1. Installation</h3>
<div class="bd-anchor" id="1-installation-2"></div>
<p>Zaloha2 is a small and simple directory synchronizer in a form of a Bash script. We can download it from its <a href="https://github.com/Fitus/Zaloha2.sh">GitHub repository</a>.</p>
<p>Once we have downloaded the project, we&#8217;ll get the following package:</p>
<pre><code class="language-bash">$ tree

├── Simple_Demo_step6.sh

$ ls -la

<p>As we can see for the above <em>ls</em> output, <em>Zaloha2.sh</em> now has the <em>x</em> permission. As a result, we can execute the script file as a program or script.</p>
<h3 id="bd-2-usage-2" data-id="2-usage-2">4.2. Usage</h3>
<div class="bd-anchor" id="2-usage-2"></div>
<p>Using Zaloha2 is pretty straightforward. First, let&#8217;s say we have the following directory structure:</p><div class="code-block code-block-6" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__336x280 __336x280" id="baeldung_incontent_2">
<script data-cfasync="false" type="text/javascript">
    freestar.config.enabled_slots.push({ placementName: "baeldung_incontent_2", slotId: "baeldung_incontent_2" });
  </script>
</div></div>
<pre><code class="language-bash">$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   ├── file03.bin
│   ├── subdir1
│   │   └── file04.bin
│   └── subdir2
│       └── file05.bin

        └── file0E.bin

6 directories, 10 files</code></pre>
<p>We then run the sync process:</p>
<pre><code class="language-bash">$ ./Zaloha2.sh --sourceDir=&quot;/home/baeldung/source&quot; --backupDir=&quot;/home/baeldung/target&quot; --color --revUp --revNew --noRemove --sha256 --findSourceOps=&#039;( -type d -a -name subdir ) -prune -o&#039; --findSourceOps=&#039;( -type d -a -name .svn ) -prune -o&#039; --findSourceOps=&#039;( -type d -a -name .cache ) -prune -o&#039; --findGeneralOps=
...
Execute above listed copies to /home/baeldung/target/ ?
[Y/y=Yes, S/s=do nothing and show further, other=do nothing and abort]: y
...
$
$ tree
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   ├── file03.bin
│   ├── subdir1
│   │   └── file04.bin
│   └── subdir2
│       └── file05.bin
└── target
    ├── file01.bin
    ├── file02.bin
    ├── file03.bin
    ├── file0A.bin
    ├── file0B.bin
    ├── file0C.bin
    ├── subdir1
    │   └── file04.bin
    ├── subdir2
    │   └── file05.bin
    ├── subdir3
    │   └── file0D.bin
    └── subdir4
        └── file0E.bin

8 directories, 15 files</code></pre>
<p>Let&#8217;s examine the options that we use:</p>
<ul>
<li><em>&#8211;sourceDir</em>: source directory</li>
<li><em>&#8211;backupDir</em>: target directory</li>
<li><em>&#8211;color</em>: to colorize the changes during interactive prompt</li>
<li><em>&#8211;revUp</em>: reverse-update; if there are newer objects in target, copy them to the source</li>
<li><em>&#8211;revNew</em>: reverse-new; if there are objects in the target that don&#8217;t exist in the source and they are newer than the last run of Zaloha2, copy them to the source</li>
<li><em>&#8211;noRemove</em>: do not remove objects in target; only add or update existing objects in the target</li>
<li><em>&#8211;sha256</em>: besides comparing file size and modified date, check file SHA256 checksum</li>
<li><em>&#8211;noProgress</em>: do not display messages from the analysis phase</li>
<li><em>&#8211;findSourceOps</em>: exclude certain sub directories from sync. If such sub directory exists in the target, it will be deleted</li>
<li><em>&#8211;findGeneralOps</em>: if it is empty, it will disable internally defined defaults that exclude certain directories (&#8220;trash&#8221; or &#8220;lost+found&#8221;). It can also be used instead of <em>findSourceOps</em> to generally exclude files or directory</li>
</ul>
<p>Unlike <em>rsync</em> or Unison, Zaloha2 <strong>doesn&#8217;t reverse copy files or directories from target to source initially, unless those files or directories are newer than the last run of Zaloha2</strong>.</p>
<h3 id="bd-3-reverse-copy" data-id="3-reverse-copy">4.3. Reverse Copy</h3>
<div class="bd-anchor" id="3-reverse-copy"></div>
<p>Let&#8217;s update some files in the target directory, then run Zaloha2 again to trigger reverse copy:</p>
<pre><code class="language-bash">$ echo &quot;edit&quot; &gt; target/file0B.bin 
$ echo &quot;edit&quot; &gt; target/file0C.bin 
$ echo &quot;edit&quot; &gt; target/subdir3/file0D.bin 
$
$ ./Zaloha2.sh --sourceDir=&quot;/home/baeldung/source&quot; --backupDir=&quot;/home/baeldung/target&quot; --color --revUp --revNew --noRemove --sha256 --findSourceOps=&#039;( -type d -a -name subdir ) -prune -o&#039; --findSourceOps=&#039;( -type d -a -name .svn ) -prune -o&#039; --findSourceOps=&#039;( -type d -a -name .cache ) -prune -o&#039; --findGeneralOps=
...
Execute above listed reverse-copies to /home/baeldung/source/ ?
[Y/y=Yes, S/s=do nothing and show further, other=do nothing and abort]: y
...
$
$ tree 
.
├── source
│   ├── file01.bin
│   ├── file02.bin
│   ├── file03.bin
│   ├── file0B.bin
│   ├── file0C.bin
│   ├── subdir1
│   │   └── file04.bin
│   ├── subdir2
│   │   └── file05.bin
│   └── subdir3
│       └── file0D.bin
└── target
    ├── file01.bin
    ├── file02.bin
    ├── file03.bin
    ├── file0A.bin
    ├── file0B.bin
    ├── file0C.bin
    ├── subdir1
    │   └── file04.bin
    ├── subdir2
    │   └── file05.bin
    ├── subdir3
    │   └── file0D.bin
    └── subdir4
        └── file0E.bin

9 directories, 18 files</code></pre>
<p>As we can see from the output of the <em>tree</em> command above, the changes in the target directory were copied to the source directory, though both directories are still not identical. This is because those files or directories in the target directory aren&#8217;t newer than the last run of Zaloha2.</p>
<h3 id="bd-4-non-interactive-mode" data-id="4-non-interactive-mode">4.4. Non-interactive Mode</h3>
<div class="bd-anchor" id="4-non-interactive-mode"></div>
<p>If we want to set up automatic/non-interactive mode, there are steps that are a bit complicated, albeit flexible, to implement, which the author suggests in the <a href="https://github.com/Fitus/Zaloha2.sh/blob/master/DOCUMENTATION.md">documentation</a>.</p>
<p>Firstly, we need to run the command with an extra option <em>&#8211;noExec</em>, which will only generate the script files in the <em>.Zalora_metadata</em> directory in the target directory. Then, we need to write a wrapper script to execute those scripts. Please refer to the documentation in the Automatic Operations section.</p>
<p><strong>Zaloha2 project is still in active development — use with caution</strong>.</p><div class="code-block code-block-7" style="margin: 8px 0; clear: both;">

<h2 id="bd-using-freefilesync-program" data-id="using-freefilesync-program">5. Using FreeFileSync Program</h2>
<div class="bd-anchor" id="using-freefilesync-program"></div>
<h3 id="bd-1-installation-3" data-id="1-installation-3">5.1. Installation</h3>
<div class="bd-anchor" id="1-installation-3"></div>
<p><a href="https://freefilesync.org/">FreeFileSync</a> is an open-source, multi-platform software. It has a GUI interface, but also provides a feature to do batch/non-interactive/text-based mode.</p>
<p>We can download the FreeFileSync installer script from its homepage and install it. The installer will also create the Desktop shortcut which we can use to run the program.</p>
<h3 id="bd-2-usage-3" data-id="2-usage-3">5.2. Usage</h3>
<div class="bd-anchor" id="2-usage-3"></div>
<p>This is the main UI of FreeFileSync:</p>
<img fetchpriority="high" fetchpriority="high" decoding="async" class="alignnone wp-image-63052 size-full" title="FreeFileSync" src="/wp-content/uploads/sites/2/2022/01/baeldung_freefilesync.jpg" alt="FreeFileSync" width="1081" height="680" />
<p>&nbsp;</p>
<p>The interface is quite intuitive:</p>
<ol>
<li>Change comparison settings, such as compare based on the file modified time and size, or file content, or file size.</li>
<li>Start comparison between the two directories.</li>
<li>Filter: we can include or exclude certain files or directories based on the file name, extension, size, or modified time.</li>
<li>Change synchronization settings: we can synchronize in two directions (Two way), one-way (Mirror), update only (Update), or custom.</li>
<li>Start synchronization.</li>
<li>Source directory.</li>
<li>Target directory.</li>
</ol>
<h3 id="bd-3-non-interactive-mode-1" data-id="3-non-interactive-mode-1">5.3. Non-interactive Mode</h3>
<div class="bd-anchor" id="3-non-interactive-mode-1"></div>
<p>If we want to create a batch/non-interactive mode, first we need to select the source and target directories from the main UI, then click on menu <em>File &gt; Save as batch job&#8230;</em>, which will show the following dialog:</p>
<img loading="lazy" decoding="async" class="alignnone wp-image-63050 size-full" src="/wp-content/uploads/sites/2/2022/01/baeldung_freefilesync_batch.jpg" alt="FreeFileSync Batch Job" width="577" height="338" />
<p>We need to disable the <em>Progress dialog</em> option and enable the <em>Ignore errors</em> option. Otherwise, it&#8217;ll display the interactive/error message box and will wait for interaction indefinitely.</p>
<p>After we created the batch job file, e.g. <em>BatchRun.ffs_batch</em>, we need to set FreeFileSync to run it.</p><div class="code-block code-block-8" style="margin: 8px 0; clear: both;">
<div align="center" data-freestar-ad="__336x280 __336x280" id="baeldung_incontent_4">

<p>Let&#8217;s create a cron job:</p>
<pre><code class="language-bash">$ crontab -e

0 1 * * * &lt;FreeFileSync installation path&gt;/FreeFileSync &lt;batch file path&gt;/BatchRun.ffs_batch</code></pre>
<p>The cron job will run the sync every day at 1AM.</p>

<h3 id="bd-2-benchmark" data-id="2-benchmark">6.2. Benchmark</h3>
<div class="bd-anchor" id="2-benchmark"></div>
<p>We measured the performance of each command on the following system and data:</p>
<ul>
<li>Dell Latitude Intel Core i7-6600U CPU @ 2.60GHz × 4, 16GB RAM</li>
<li>Source directory: Seagate external HDD 5TB, 7200rpm, SATA III, ext4</li>
<li>Target directory: Seagate external HDD 1.5TB, 7200rpm, SATA III, NTFS</li>
<li>Data synchronized: 10GB, 11,780 image files</li>
<li>OS: Debian Buster</li>
</ul>
<p>The number of files and total size of the source directory:</p>
<pre><code class="language-bash">$ find source -type f | wc -l
11780
$
$ du source -hc --max-depth=0
1.3G source/subdir1
3.3G source/subdir2
5.5G source/subdir3
10G source
10G total</code></pre>
<p>For the test, we <strong>synced both directories three times</strong>. We ran each command for three times, except for <em>rsync</em><strong><em>. </em></strong>We <strong>ran <em>rsync</em> six times &#8211; twice for each sync &#8211; because it is not bidirectional</strong>:</p>
<ul>
<li>for the initial sync, we synced/mirrored data between <em>source</em> and <em>target</em> directories</li>
<li>for the second sync, all tools returned immediately as there&#8217;s nothing to sync</li>
<li>finally for the third sync, first we deleted some data in <em>source</em> (delete <em>subdir1</em>), then added some data in <em>target</em> (copy <em>subdir1</em> and name it <em>subdir4</em>), then we synced the directories.</li>
</ul>
<p>The options that we used for each command:</p>
<ul>
<li><em>rsync</em>
<pre><code class="language-bash">$ rsync -PScauv --info=progress2 source/ /media/baeldung/target/
$ rsync -PScauv --info=progress2 target/ /media/baeldung/source/</code></pre>
</li>
<li>Unison
<pre><code class="language-bash">$ ./unison -batch=true /media/baeldung/source/ /media/baeldung/target/</code></pre>
</li>
<li>Zaloha2
<pre><code class="language-bash">$ ./Zaloha2.sh --sourceDir=&quot;/media/baeldung/source&quot; --backupDir=&quot;/media/baeldung/target&quot; --color --revUp --revNew --noRemove --noRestore --mawk --findGeneralOps=</code></pre>
</li>
<li>FreeFileSync
<pre><code class="language-bash">$ which FreeFileSync
/home/baeldung/.local/bin/FreeFileSync
$ ./home/baeldung/.local/bin/FreeFileSync BatchRun.ffs_batch</code></pre>
</li>
</ul>

<h2 id="bd-conclusion" data-id="conclusion">7. Conclusion</h2>
<div class="bd-anchor" id="conclusion"></div>
<p>Of all four tools that we tested, we found that Unison is the best for synchronizing directories in both directions.</p>
<p>Unison may take a bit longer when we run it for the first time, but the subsequent run was the fastest compared to the other tools in terms of performance. It supports conflict detection, is resilient to failure (it has revision control, useful for rollback), is bidirectional, and has a command-line as well as a GUI interface.</p>
<p>Coming in a close second is FreeFileSync, which is also great with its intuitive GUI.</p>
